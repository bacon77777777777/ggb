# 吉吉比（GGB）開發時程計畫書

> 版本 v1.0　｜　文件日期：2026-07-07　｜　讀者：四位股東 ＋ AI 團隊
> 對應願景：前期以線上轉蛋為核心，穩定後擴展卡牌、PSA 收藏品、玩家交易、拍賣等娛樂服務。

---

## 一、開發原則

1. **每個功能獨立可測試**；涉及金額的邏輯一律保留完整明細供人工複核。
2. **完成一個階段再統一推版**，避免觸發 Vercel 每日部署限制。
3. **先穩後擴**：先把首發核心（轉蛋＋金流＋自動化）做穩，再談新品類。
4. **AI 優先降本**：能用 cron agent / GB哥 / AI 內容自動化的，就不增加人力。
5. **資料驅動**：以留存、對帳、風控數據決定下一步，而非憑感覺。

---

## 二、開發階段路線圖（Roadmap）

| 階段 | 期間（規劃） | 主題 | 產出 |
|---|---|---|---|
| P0 首發收尾 | 2026-07（7/07–7/27） | 補齊首發必要項、測試、正式金流 | 可上線的轉蛋平台 |
| P1 首發與穩定 | 2026-08 | 正式首發、密集監控與修補 | 穩定營運、首月月結 |
| P2 留存與流量 | 2026-09 – 2026-10（Q3–Q4） | 留存機制、自然流量獲客、體驗優化 | 可複製的獲客模式 |
| P3 品類擴展一 | 2026-11 – 2026-12（Q4） | 卡牌抽獎深化 / 玩家交易強化 | 新品類上線其一 |
| P4 品類擴展二 | 2027 H1 | PSA 收藏品、拍賣功能 | 高價收藏與拍賣 |
| P5 平台化 | 2027 H2+ | 開放更多娛樂服務、規模化 | 娛樂平台雛形 |

## 三、各階段詳細規劃

### P0 首發收尾（2026 年 7 月）

聚焦「上線前必要項」，細節與 Gate 見《03 上線時程規劃表》。核心開發任務：

- 轉蛋殺率動態化（drawLogic 讀 `product_settings.current_profit_rate`）。
- 退款人工 SOP 與後台提示（ECPay 退款 API 留待 P1）。
- 前台 P0 E2E（金流、抽獎、代幣一致性）。
- 後台 P0 人工驗收（財務、GB哥、物流）。
- 首發商品建檔與圖文條碼。

負責：我＋施（開發）、楊（商品／出納）、陳（行銷素材）。

### P1 首發與穩定（2026 年 8 月）

- **正式首發**（M1）與行銷同步啟動。
- **退款 API 整合**：串接 ECPay 退款，取代人工操作。
- **對帳自動化強化**：reconcile-payments 異常通知完善。
- **廠商月結自動付款通知**：月結快照 → 自動通知應付。
- **物流「已送達」異常確認**：驗證 auto-deliver 實際運作。
- 修補首發後回報的 bug（快速迭代）。

負責：我＋施（開發）、楊（月結／對帳）、陳（客訴／客戶關係）。

### P2 留存與流量（2026 年 9–10 月）

**留存與體驗**

- 轉蛋機模組化：機台區塊獨立為可替換模組，設計多套主題（東洋扭蛋機、夾娃娃機、街機風格），後台可依商品切換，強化視覺差異化。
- 代幣流水帳前後餘額展示頁優化。
- 任務／簽到／排行榜活動化，配合節慶檔期。
- 客服 Agent 能力擴充（多步驟退款、商品詢問），降低轉人工比例。

**自然流量獲客（低成本、可複製）**

- 用 generate-content cron 量產 Reels/Shorts/Threads 素材，建立內容生產線。
- 會員推薦機制上線（推薦碼、雙向獎勵）。
- SEO、新聞稿、KOL 與異業合作導流，記錄各渠道 ROI 找出可放大的模式。

負責：我（模組化／體驗）、施（客服 Agent／維護）、陳（KOL／異業／推薦機制）、楊（活動商品）。

### P3 品類擴展一（2026 年 11–12 月）

依 P2 數據選擇先做**卡牌抽獎深化**或**玩家交易強化**其一：

- 卡牌抽獎：集卡進度、稀有度、卡冊收藏、限定卡池。
- 玩家交易：市集與交換體驗升級、交易安全（託管、糾紛處理）、交易手續費機制。

負責：我＋施（開發）、陳（交易市場推廣）、楊（供貨與定價）。

### P4 品類擴展二（2027 H1）

- **PSA 高價收藏品**：鑑定分級商品上架、真偽與保值資訊、高單價金流與物流保障。
- **拍賣功能**：競標、出價、結標與付款流程、防作弊風控。

負責：全體，依主責分工並跨域補位。

### P5 平台化（2027 H2 起）

在會員、流量、數據累積到一定規模後，逐步開放更多娛樂相關服務，朝「娛樂平台」定位擴張。具體項目依當時市場與數據再由四人共識決定。

## 四、Sprint 節奏建議

- 以**兩週為一個 Sprint**，每個 Sprint 有明確可測試產出。
- Sprint 開始：四人快速對齊本期目標與主責；Sprint 結束：Demo ＋ 回顧，統一推版。
- 涉及金額的功能，Sprint 內必含對應測試案例（對照《02 前後台測試案例》）。
- 缺功能由 GB哥 `log_capability_gap` 記錄，進 backlog 排期。

---

## 五、資源與成本規劃

**人力**：前期無員工，四位股東即核心成員，前期不支薪，資金優先投入平台、商品、系統與營運。

**基礎設施成本**（依會員規模分階段）：

| 階段 | 會員規模 | 月成本 |
|---|---|---|
| 現況 | 0–500 | $0（全 Free 方案） |
| 成長期 | 500–2,000 | 約 $45（Supabase Pro $25 + Vercel Pro $20） |
| 高峰期 | 單日 UV >1,000 | 視活動另評估（含 Supabase 連線提升 / Team 方案） |

**外部服務**：ECPay（金流＋物流手續費）、LINE Messaging API（免費額度內）、Anthropic API（GB哥＋cron agents 用量，隨規模成長）。

## 六、風險與相依

| 風險 | 對應階段 | 緩解 |
|---|---|---|
| 退款 API 未整合 | P0→P1 | P0 用人工 SOP，P1 完成整合 |
| 後台無自動化測試 | P0 起 | 逐步導入 API 整合測試，優先覆蓋金額路徑 |
| 流量獲客未驗證 | P2 | 多渠道小規模試打，用數據找可放大模式 |
| 新品類金流／物流複雜（PSA／拍賣） | P4 | 提前研究高單價交易保障與風控 |
| 預算有限 | 全程 | 自然流量優先、AI 自動化降本、分階段擴容 |

---

## 七、成功指標（KPI，隨階段調整）

- **P1**：首發後兩週零重大事故、對帳 100% 一致、首月完成月結。
- **P2**：找到至少一個「成本低、可複製」的獲客渠道；次日／7 日留存達目標。
- **P3–P4**：新品類貢獻營收占比、交易活躍度、客單價提升。
- **全程**：GB哥＋cron agents 讓日常營運維持「一則早報即可掌握」的自動化程度。
