## 一、專案總覽

吉吉比（GGB，吉吉比有限公司 / GGB CO., LTD.）是一個以**線上轉蛋**為核心的娛樂平台。我們要打造的不是一間「賣抽獎商品的店」，而是一個能持續累積會員、品牌、流量與數據的娛樂平台。前期以線上轉蛋（一番賞、盒玩、轉蛋、抽卡）切入，平台穩定後逐步擴展到卡牌抽獎、PSA 高價收藏品、玩家交易、拍賣等更多娛樂服務。

技術上，GGB 由**兩個獨立的 Next.js 15 應用**組成，共用同一個 Supabase（PostgreSQL）資料庫：

| 應用 | 對象 | 內容 |
|---|---|---|
| `frontend/` | 玩家端（PWA） | 轉蛋機、盲盒、卡牌、商店、倉庫、儲值、簽到、任務、排行榜、二手市集、交換、公平性驗證 |
| `backend/` | 後台管理 | 商品/訂單/廠商/財務/風控/AI 組織，以及完整的自動化「神經系統」 |

平台的一大特色是**AI 自動化組織**：由 16 個排程代理人（cron agents）＋ LINE 智能管家「GB哥」組成，自動盯營收、風控、供應鏈、行銷與技術運維，異常才通知人。目標是讓經營團隊從「操作者」變成「看數據做決策的老闆」。

## 二、產品願景與策略（與創業共識一致）

**核心信念**：商品人人能賣、無法獨佔；真正有價值的是平台、會員、流量與數據。

**發展路線**：

1. 第一階段 — 線上轉蛋為核心，累積會員、品牌與自然流量。
2. 第二階段 — 擴展卡牌抽獎、PSA 高價收藏品、玩家交易平台、拍賣功能、更多收藏娛樂商品。

**流量策略**：最大限制是預算而非能力，前期不重壓廣告投放，改以低成本、可複製的自然流量為主 — Reels / Shorts / Threads、社群經營、SEO、媒體曝光與新聞稿、KOL 合作、會員推薦機制、異業合作。找到「成本低、效果好、可持續複製」的獲客模式後再逐步放大。

平台的技術設計已經呼應此策略：內建公平性驗證（建立信任）、排行榜與任務簽到（提升留存）、二手市集與交換（增加互動與黏著）、AI 內容草稿生成（降低社群經營成本）、競品情報爬取（掌握市場）。

## 三、系統架構

### 3.1 兩個獨立應用

**Frontend（玩家端 PWA）** 以 Supabase Auth 做身份驗證（`@supabase/ssr`），玩家登入後 cookie 由 middleware 管理。核心 Context 包含 `AuthContext`（session 狀態）與 `FeatureFlagsContext`（功能開關，可一鍵開關轉蛋等功能）。

**Backend（管理後台）** 不使用 Supabase Auth，改用自製的 JWT-like token（`lib/adminSession.ts`）：HMAC-SHA256 簽章存在 `admin_session` cookie，每日台灣午夜到期，所有後台 API 都呼叫 `requireAdminSession()` 驗證，並以 `logAdminAction()` 寫入稽核軌跡。

### 3.2 後台關鍵模組（lib）

| 檔案 | 用途 |
|---|---|
| `supabaseAdmin.ts` | service role client，繞過 RLS，後台所有寫入都用它 |
| `adminSession.ts` / `requireAdmin.ts` | 管理員 session 簽章、驗證 |
| `logAdminAction.ts` | 稽核軌跡（`action_logs`）＋ 取 client IP |
| `gbBro.ts` | GB哥 LINE AI 助手（Claude Haiku + tool loop，27 個工具） |
| `csAgent.ts` | 客服 AI（前台玩家 LINE 訊息） |
| `ecpay.ts` / `ecpay_logistics.ts` | 綠界金流（SHA256 簽章）／物流（MD5 簽章） |
| `webhookIdempotency.ts` | ECPay callback 冪等性防護（`webhook_events` 表） |
| `financeMetrics.ts` | 財務指標計算 |

### 3.3 資料庫核心設計

- **`token_ledger` 是 VIEW（非實體表）**，UNION ALL 四個來源：`recharge_records`（recharge / marketing / test）、`draw_records`（draw / dismantle）、`token_adjustments`（manual）。手動補幣一律寫 `token_adjustments`，**不可**寫 `recharge_records`（後者是 ECPay 對帳基礎）。
- **機器人排除**：所有財務／分析 query 必須加 `WHERE (is_bot IS NULL OR is_bot = false)`。
- **公平性**：`draw_records.txid_hash` 讓玩家可自行重算驗證抽獎結果。
- **`execute_readonly_sql` RPC**：GB哥與 cron agent 查詢用，僅允許 SELECT / WITH，由 service_role 呼叫。
- **Migrations**：`backend/db/migrations/<n>_name.sql`，編號遞增，目前已累積至 287，直接以 psql 執行。

### 3.4 技術棧

前台：Next.js 15.5、React 18、Tailwind CSS、Framer Motion、Lucide、Supabase SSR、Upstash Redis（rate limit）、Sentry、Playwright（E2E）。
後台：Next.js 15.5、Anthropic SDK（Claude Haiku）、@line/bot-sdk、pg 直連、Sharp（圖片）、date-fns、PapaParse、Sentry。
部署：前後台皆部署於 Vercel；GitHub Actions 負責定期備份與 migration 執行。

> 另有 risk-check、competitive-intel、ai-cto 等 agent。所有跨部門信號會 INSERT `agent_events` 並推 LINE，於後台事件中心集中顯示。

### 4.4 GB哥（LINE 智能管家）

入口 `api/line/webhook`，核心 `lib/gbBro.ts`。群組需含「gb哥」觸發、個人訊息由 `ADMIN_LINE_IDS` 白名單控管；30 分鐘對話記憶、最多 5 輪 tool loop。**執行原則：收到指令立即執行回報，絕不把問題丟回給老闆。** 提供 27 個工具，涵蓋查詢（營收、待辦、庫存、訂單、會員、月結、退款、任意唯讀 SQL）與寫入（改庫存／價格／上下架、補幣、凍結／標記帳號、標記送達、更新追蹤號、取消訂單、核准退款、建立折扣碼、更新文案、記錄功能缺口）。

### 4.5 金流與物流（ECPay）

金流：`topup` → 建立 `recharge_records`(pending) → 綠界 AioCheckOut V5 → callback 驗簽（SHA256）＋冪等性檢查 → 補 G 幣。物流：後台建立物流單 → CVS 地圖選店／HOME 宅配 → callback（MD5）更新 `tracking_number` 與訂單狀態。冪等性由 `webhook_events` 表保證同一筆交易只入帳一次。

## 五、角色分工（對應四位股東）

以下為依各自專長規劃的**主責方向**，非固定分工。公司前期沒有員工，每位股東都需一起投入，也歡迎跨領域互相補位。主責是為了提升效率，不是劃清界線。

| 股東 | 主責 | 對應到平台的模組 |
|---|---|---|
| **我** | 平台開發、AI 應用、視覺設計、產品規劃、商品上架 | frontend/backend 開發、AI 組織與 GB哥、UI/視覺、產品 roadmap、商品建檔 |
| **楊** | 商品管理、廠商合作、出納、供應鏈管理 | products / suppliers / settlement 對接、廠商月結、出納對帳、supply-chain 訊號跟進 |
| **施** | 平台開發、AI 工具應用、程式協助、平台維護、商品上架 | frontend/backend 開發、cron agents 維護、bug 修復、商品建檔 |
| **陳** | 業務開發、對外合作、商務洽談、客戶關係經營 | KOL／異業合作、廠商與通路洽談、社群與客戶關係、流量策略落地 |

**協作原則（兄弟公約精神）**：有話直接說、尊重彼此、有功一起分享有責一起扛、有人累了就說出來、不分你我互相補位、誠信透明。有人忙就有人多幫一點，公司的成功是四個人一起努力的成果。

## 六、決策與治理機制

**重大決策由四位股東共同討論並取得共識**，包括：增資／減資、股權調整、新股東加入、利潤分配、重大合作案、大額支出、公司方向重大調整等。公司沒有誰說了算。

**制度建立（成立初期即完成）**：股東協議、股權比例、退場機制、股份轉讓限制、保密條款、財務透明、權利義務、負責人權責。制度不是因為不信任，而是相信大家會一起走很久，所以更需要先把規則建立好。

**帳務透明**：收入、支出與帳務交由專業記帳士處理，所有股東都能了解營運與財務狀況。平台本身的 `token_ledger`、月結快照、稽核軌跡（`action_logs`）也提供技術面的透明基礎。

**公司設立**：資本額預計 20 萬元，四位股東共同出資；負責人由共同討論決定（若由家人掛名，同步完成股東協議與法律文件）。前期不支薪，資金優先投入平台、商品、系統與營運，待穩定獲利後再討論薪酬與分紅。

## 七、風險管理

| 風險 | 影響 | 對策 | 主責 |
|---|---|---|---|
| ECPay 尚為測試帳號 | 無法收真實金流 | 申請正式帳號，切換 Merchant/Hash 環境變數（見上線時程規劃表） | 我／楊 |
| 退款 API 未整合 | 退款需人工至綠界操作 | 上線後補 ECPay 退款 API 整合，短期以人工 SOP 補位 | 我／施 |
| 後台無自動化測試 | 改動易造成回歸錯誤 | 依測試案例文件補齊後台核心流程測試 | 施 |
| 轉蛋殺率未從 DB 動態讀取 | 無法即時調整獲利率 | 讓 drawLogic 讀取 `product_settings.current_profit_rate` | 我 |
| 預算有限 | 無法大量投放廣告 | 走自然流量策略，AI 內容草稿降低成本 | 陳／我 |
| 流量成長造成 DB 連線瓶頸 | 尖峰時段錯誤 | 分階段升級 Supabase/Vercel Pro，必要時 feature flag 關閉轉蛋 | 施 |
| 競品情報依賴外部 API | 報告可能不完整 | 加入備援爬取源與重試邏輯 | 施 |
| 合夥人認知落差 | 影響合作與友誼 | 共識會議＋股東協議＋定期溝通 | 全體 |
