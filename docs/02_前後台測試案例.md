# 吉吉比（GGB）前後台測試案例（Test Cases）

> 版本 v1.0　｜　文件日期：2026-07-07　｜　讀者：AI 團隊 ＋ 股東
> 用途：**首發前驗收基準**。前台以 Playwright E2E 補齊為目標；後台目前無自動化測試，先以人工驗收，再逐步自動化。

---

## 使用說明

- **優先級**：P0 = 上線前必過（金流、抽獎、代幣正確性）；P1 = 重要體驗；P2 = 加分項。
- **狀態欄**：測試時填 ✅ 通過 / ❌ 失敗 / ⬜ 未測。
- **測試環境**：ECPay 使用測試帳號；資料庫使用 seed 資料（`reset_and_seed.sql`）；測試帳號請排除或標記 `is_bot`，避免污染財務報表。
- 每個案例格式：前置條件 → 步驟 → 預期結果。

---

# 第一部分：前台（玩家端）

## A. 帳號與認證

### TC-F-A01｜註冊新帳號（P0）
- 前置：未登入。
- 步驟：進入 `/register` → 輸入 email/密碼 → 送出 → 收驗證信 → 完成驗證。
- 預期：帳號建立成功，`users` 新增一筆，初始 G 幣為 0（或依 seed 設定），可登入。

### TC-F-A02｜登入 / 登出（P0）
- 步驟：`/login` 輸入正確帳密登入；再登出。
- 預期：登入後 session cookie 建立、可見個人資料；登出後受保護頁面導回登入。

### TC-F-A03｜錯誤密碼與忘記密碼（P1）
- 步驟：輸入錯誤密碼；再走 `/forgot-password` → 收信 → `/update-password` 改密碼。
- 預期：錯誤密碼顯示錯誤訊息、不登入；改密碼後可用新密碼登入。

## B. 儲值與金流（P0，重點）

### TC-F-B01｜儲值下單跳轉綠界
- 前置：已登入。
- 步驟：`/topup` → 選金額 → 前往結帳。
- 預期：建立 `recharge_records`(status=pending)，正確跳轉綠界 AioCheckOut V5，CheckMacValue 驗證通過。

### TC-F-B02｜付款成功補幣
- 步驟：於綠界測試環境完成付款 → 回調 callback。
- 預期：`recharge_records` 轉 success、G 幣正確增加、`token_ledger` 出現 type=recharge 一筆、前台餘額即時更新。

### TC-F-B03｜callback 冪等性（重複回調）
- 步驟：以同一 `MerchantTradeNo` 觸發 callback 兩次。
- 預期：只補一次幣，第二次記為 duplicate 仍回傳成功，`webhook_events` 有紀錄。

### TC-F-B04｜付款失敗 / 取消
- 步驟：在綠界取消付款或模擬失敗。
- 預期：`recharge_records` 不轉 success、不補幣、前台顯示失敗結果。

## C. 轉蛋 / 盲盒 / 卡牌（P0，重點）

### TC-F-C01｜G 幣足夠抽獎成功
- 前置：餘額足夠。
- 步驟：進入某商品 `/gacha/[id]`（或 blindbox/card）→ 抽一次。
- 預期：扣正確 G 幣、`draw_records` 新增（含 prize、txid_hash）、獎品進入倉庫、獎項 remaining 減 1。

### TC-F-C02｜G 幣不足阻擋
- 前置：餘額不足。
- 步驟：嘗試抽獎。
- 預期：阻擋並提示儲值，不扣幣、不產生 draw_records。

### TC-F-C03｜獎項售罄
- 前置：某獎項 remaining=0。
- 步驟：抽該商品。
- 預期：不抽出已售罄獎項；商品全數售罄時商品標記 sold_out、不可再抽。

### TC-F-C04｜公平性驗證
- 步驟：抽獎後至 `/fairness/[id]` 用 txid_hash 驗算。
- 預期：顯示可驗證資訊，重算結果與紀錄一致。

### TC-F-C05｜連續／批量抽獎代幣一致性（P0）
- 步驟：連抽多次。
- 預期：扣幣總額 = 單抽 × 次數；`token_ledger` 消耗筆數與金額一致，無漏扣或重複扣。

## D. 倉庫、拆解與配送

### TC-F-D01｜倉庫顯示物品（P1）
- 步驟：抽中後查看 `/item`。
- 預期：新物品出現，狀態正確（可出貨／可交換／可出售）。

### TC-F-D02｜拆解退還 G 幣（P1）
- 步驟：對可拆解物品執行拆解。
- 預期：物品消失、退還對應 G 幣、`token_ledger` 出現 type=dismantle。

### TC-F-D03｜申請出貨與物流選擇（P0）
- 步驟：選物品申請出貨 → 選 CVS 超商（地圖選店）或 HOME 宅配 → 送出。
- 預期：建立 `orders`、物品鎖定、CVS 正確帶回門市代碼。

### TC-F-D04｜訂單狀態追蹤（P1）
- 步驟：`/purchases` 查看訂單。
- 預期：狀態隨物流更新（submitted → shipping → delivered），顯示追蹤號。

## E. 社交與留存

### TC-F-E01｜每日簽到（P1）
- 步驟：`/check-in` 簽到，隔日再簽。
- 預期：發對應獎勵、同日不可重複簽、連續天數正確。

### TC-F-E02｜任務完成與領獎（P1）
- 步驟：`/mission` 完成任務條件後領獎。
- 預期：任務進度正確追蹤、達標可領、獎勵入帳（對照 migration 278 任務追蹤修正）。

### TC-F-E03｜排行榜顯示（P2）
- 步驟：`/ranking` 查看。
- 預期：排名正確，機器人帳號依設定顯示，真實用戶統計排除 is_bot。

## F. 二手市集與交換

### TC-F-F01｜上架二手物品（P1）
- 步驟：`/sell/new` 上架 → `/sell/manage` 管理。
- 預期：上架成功、市集 `/market` 可見、可下架。

### TC-F-F02｜購買二手物品（P1）
- 步驟：於市集購買他人物品。
- 預期：建立 sell-order、G 幣／物品正確移轉、雙方收到訊息。

### TC-F-F03｜發起交換（P2）
- 步驟：`/exchange/new` 發起 → 對方回應。
- 預期：交換流程與物品移轉正確、產生 exchange-order。

## G. PWA / 導航（既有 E2E）

### TC-F-G01｜頁籤導航（P1，已有 navigation.spec.ts）
- 預期：首頁 → 交換 → 排行榜 導航正常。

### TC-F-G02｜PWA 輸入框焦點（P2，已有 pwa-input-focus.spec.ts）
- 預期：PWA 模式下輸入框聚焦不跑版。

# 第二部分：後台（管理端）

## H. 後台認證與權限

### TC-B-H01｜管理員登入 / session（P0）
- 步驟：`/login` 登入 → 存取受保護頁面。
- 預期：`admin_session` cookie 建立、`requireAdminSession()` 通過；午夜台灣時間到期後需重新登入。

### TC-B-H02｜權限分級（P1）
- 步驟：以不同角色登入存取受限模組（permissions 設定）。
- 預期：無權限者被擋、有權限者可操作。

### TC-B-H03｜稽核軌跡（P0）
- 步驟：任一寫入操作後查 `/logs`。
- 預期：`action_logs` 記錄操作者、動作、client IP、時間。

## I. 商品與庫存管理

### TC-B-I01｜新增商品與獎項（P0）
- 步驟：`/products/new` 建立商品、設定獎項機率與數量。
- 預期：`products` 與 `product_prizes` 正確寫入，機率總和合理。

### TC-B-I02｜商品上下架（P0）
- 步驟：改商品狀態 active / archived / sold_out。
- 預期：狀態變更即時反映前台可見性。

### TC-B-I03｜批量上架與條碼（P1）
- 步驟：批量操作、掃描／輸入 barcode（對照 migration 285）。
- 預期：批次成功、條碼正確儲存與顯示。

### TC-B-I04｜庫存調整（P1）
- 步驟：調整 remaining（正負 delta）。
- 預期：庫存正確增減，低於門檻觸發 supply-chain 警示。

## J. 訂單與物流

### TC-B-J01｜訂單列表與詳情（P1）
- 步驟：`/orders` 查看、進入詳情。
- 預期：訂單資料、商品項目、金額正確。

### TC-B-J02｜建立物流單（P0）
- 步驟：對訂單建立 CVS / HOME 物流單。
- 預期：呼叫綠界物流（MD5 簽章）、取得物流單號、狀態轉 shipping。

### TC-B-J03｜物流 callback 更新（P0）
- 步驟：模擬綠界物流回調。
- 預期：`tracking_number` 更新、狀態同步、`webhook_events` 記錄、冪等。

### TC-B-J04｜超時自動送達（P1）
- 步驟：讓 HOME 訂單超過 7 天 / CVS 超過 3 天 → 執行 auto-deliver cron。
- 預期：自動標記 delivered，不卡單。

## K. 財務、儲值審核與退款（P0，重點）

### TC-B-K01｜代幣流水帳正確性（P0）
- 步驟：`/token-ledger` 檢視某用戶。
- 預期：recharge + marketing + manual − draw − dismantle 前後餘額連續、無跳號。

### TC-B-K02｜手動補幣寫對表（P0）
- 步驟：以後台／GB哥手動補幣。
- 預期：寫入 `token_adjustments`（type=manual），**不寫** `recharge_records`，不影響 ECPay 對帳。

### TC-B-K03｜待複核儲值處理（P1）
- 步驟：`/recharge-review` 對 pending 超時儲值確認或忽略。
- 預期：確認補幣正確；忽略記錄狀態；對照 flag-pending-recharge cron。

### TC-B-K04｜退款申請審核（P0）
- 步驟：`/refund-requests` 核准 / 拒絕。
- 預期：狀態正確更新、G 幣或金額處理正確。
- **已知限制**：實際退款（退錢到卡）ECPay API 未整合，現階段需人工至綠界操作 — 測試需驗證此 SOP 有被提示。

### TC-B-K05｜對帳公式驗證（P0）
- 步驟：對照 `expected = recharge_total + manual_total − draw_total − refund_deducted`。
- 預期：cfo-agent 對帳結果與後台一致，誤差在容許範圍（50 G/用戶）內。

## L. 廠商與月結

### TC-B-L01｜廠商建檔（P1）
- 步驟：`/suppliers` 新增廠商。
- 預期：資料、稅號、is_active 正確。

### TC-B-L02｜月結快照與分潤（P0）
- 步驟：執行 monthly-settlement → `/settlement-snapshots` 檢視。
- 預期：平台 30% / 廠商 70% 計算正確，status=draft 等人工確認，可轉 confirmed / paid。

## M. AI 組織與 GB哥

### TC-B-M01｜每日早報推播（P1）
- 步驟：手動觸發 daily-report。
- 預期：LINE 收到早報，數字與後台一致；營收下滑 >20% 有警示。

### TC-B-M02｜事件中心（P1）
- 步驟：觸發 risk-scan / supply-chain 產生事件。
- 預期：`agent_events` 新增、推 LINE、`/agent-events` 顯示待處理。

### TC-B-M03｜GB哥查詢類（P0）
- 步驟：LINE 私訊「昨天營收多少」「有什麼要處理的」。
- 預期：回覆數字與後台一致，不把問題丟回。

### TC-B-M04｜GB哥寫入類（P0）
- 步驟：LINE 指令「把某商品下架」「補某用戶 100 G」「核准這筆退款」。
- 預期：立即執行並回報，資料庫正確變更，補幣寫 `token_adjustments`，操作留稽核軌跡。

### TC-B-M05｜GB哥權限控管（P0）
- 步驟：以非 `ADMIN_LINE_IDS` 帳號私訊、群組未含「gb哥」關鍵字。
- 預期：非授權者無法操作、群組不誤觸發。

### TC-B-M06｜唯讀 SQL 安全性（P0）
- 步驟：讓 GB哥 run_sql 執行含 UPDATE/DELETE 的語句。
- 預期：被 `execute_readonly_sql` 擋下，僅允許 SELECT/WITH（對照 migration 286/287 權限修復）。

## N. 風控

### TC-B-N01｜異常抽獎偵測（P1）
- 步驟：單帳號 24h 抽獎 >80 次。
- 預期：risk-scan 標記可疑並發事件。

### TC-B-N02｜大額儲值偵測（P1）
- 步驟：單筆 > NT$5000 或日額 > NT$10000。
- 預期：觸發警示 / 標記。

### TC-B-N03｜凍結 / 解凍帳號（P1）
- 步驟：後台或 GB哥凍結可疑帳號再解凍。
- 預期：凍結後該帳號無法抽獎／交易，解凍後恢復。

### TC-X-01｜機器人排除（P0）
- 所有財務／統計報表過濾 `is_bot`，數字不含測試/機器人帳號。

### TC-X-02｜環境變數與正式金流切換（P0）
- 切換 ECPay 正式 Merchant/Hash/API_URL 後，重跑 B 組金流案例。

### TC-X-03｜負載與連線（P1）
- 尖峰模擬（k6），觀察 Supabase 連線數與 Sentry timeout，驗證 feature flag 可一鍵關閉轉蛋。

### TC-X-04｜錯誤監控（P1）
- Sentry 能正確收到前後台錯誤。

### TC-X-05｜資料備份（P1）
- GitHub Actions 備份正常，能還原。

## 附錄：自動化建議

首發前建議至少將以下 P0 案例自動化：前台 B（金流）、C（抽獎與代幣一致性）為 Playwright E2E；後台 K（財務）、M（GB哥）為 API 層測試。後台目前無測試框架，建議導入 API 整合測試（可用 Playwright request 或 vitest + supertest），優先覆蓋涉及金額與代幣的路徑。
