## 一、專案總覽

吉吉比（GGB）／吉吉比有限公司 / GGB CO., LTD.，是一個以**線上轉蛋**為核心的全平台。我們要打造的不是一間「賣實體商品的商店」，而是一個以轉蛋為累積黏著、服務、流量與數據的全平台。初期以線上轉蛋（一番賞、盲盒、轉蛋、抽卡）分類，平台穩定後逐步擴展到卡片鑑定、PSA 高端收藏服務、玩家交易、寄賣等更多全端服務。

技術上，GGB 由**兩個獨立的 Next.js 15 應用**組成，共用同一個 Supabase（PostgreSQL）資料庫：

| 應用 | 對象 | 內容 |
|---|---|---|
| `frontend/` | 玩家端（PWA） | 轉蛋機、扭蛋、盲盒、抽卡、商品庫、倉庫、儲值、簽到、任務、排行榜、二手市場、交換、公平性驗證 |
| `backend/` | 後台管理 | 商品/設定/廠商/財務/風控/AI 組織，以及完整的自動化、智慧通報系統 |

平台的三大特色是**AI 自動化組織**，由 16 個智能代理人（cron agents）、LINE 智能管家（GB哥）組成，自動監看收益、風控、供應鏈、行銷運營、競品分析，異常立即通知人。目標是讓當班負責人從「看報表、下決策」變成「看數據分析決策的薪資」。

## 二、產品理念與策略（創辦人共識一致）

**核心信念**：每個人都能玩，無法用「作弊」，真正有價值的是平台、服務、流量與數據。

**發展路線**：

1. 第一階段 → 線上轉蛋為核心，累積黏著、服務與自然流量。
2. 第二階段 → 擴展卡片鑑定、PSA 高端收藏服務、玩家交易平台、寄賣功能、更多收藏全端商品。

**流量策略**：最大限制是預算與軟體能力，初期一邊宣傳一邊開放，方以低成本、可複製的自然流量為主 — Reels / Shorts / Threads、社群經營、SEO、媒體文章撰稿、KOL 合作、客戶推薦機制、異業合作、差異化、成本最低、成效最好、可為複製、零成本模式從後退步放大。

平台的技術設計已順應此策略：內建公平性驗證（建立信任）、排行榜與任務簽到（增加玩家黏性）、二手市場與交換（增加互動與留存）、AI 內容撰稿生成（降低社群經營成本）、競品情報日報（掌握市場）。

## 三、系統架構

### 3.1 兩個獨立應用

**Frontend（玩家端 PWA）** 以 Supabase Auth 做身份驗證（`@supabase/ssr`），玩家登入後 cookie 由 middleware 管理。核心 Context 包含 `AuthContext`（session 狀態）與 `FeatureFlagsContext`（功能開關，可一鍵停用轉蛋等功能）。

**Backend（管理後台）** 不使用 Supabase Auth，改用自製 JWT-like token（`lib/adminSession.ts`），HMAC-SHA256 簽章存在 `admin_session` cookie，每日台灣午夜到期。所有後台 API 都呼叫 `requireAdminSession()` 驗證，並以 `logAdminAction()` 寫入稽核軌跡。

### 3.2 後台關鍵模組（lib）

| 檔案 | 用途 |
|---|---|
| `supabaseAdmin.ts` | service role client，繞過 RLS，後台所有寫入都用它 |
| `adminSession.ts` / `requireAdmin.ts` | 管理員 session 簽章、驗證 |
| `logAdminAction.ts` | 稽核軌跡（`action_logs`）、 client IP |
| `gbBro.ts` | GB哥 LINE AI 助手（Claude Haiku + tool loop，27 個工具）|
| `csAgent.ts` | 客服 AI（回應玩家 LINE 訊息）|
| `ecpay.ts` / `ecpay_logistics.ts` | 綠界金流（SHA256 簽章）、物流（MD5 簽章）|
| `webhookIdempotency.ts` | ECPay callback 冪等性防護（`webhook_events` 表）|
| `financeMetrics.ts` | 財務指標計算 |

### 3.3 資料庫核心設計

- **`token_ledger` 是 VIEW，非實體表**，UNION ALL 四個來源：`recharge_records`（recharge / marketing / test）、`draw_records`（draw / dismantle）、`token_adjustments`（manual）。手動補幣必須寫 `token_adjustments`，**不可**寫 `recharge_records`（後者是 ECPay 對帳基礎）。
- **機器人排除**：所有財務/分析 query 必須加 `WHERE (is_bot IS NULL OR is_bot = false)` 或使用 `getRealUserIds()`。
- **公平性**：`draw_records.txid_hash` 讓玩家可自行運算驗證抽獎結果。
- **`execute_readonly_sql` RPC**：GB哥和 cron agent 查詢用此函數，僅允許 SELECT/WITH，由 service_role 呼叫。
- **Migrations**：`backend/db/migrations/<n>_name.sql`，編號遞增，目前已累積至 293，直接以 psql 執行。

### 3.4 技術棧

前台：Next.js 15.5、React 18、Tailwind CSS、Framer Motion、Lucide、Supabase SSR、Upstash Redis（rate limit）、Sentry、Playwright（E2E）。
後台：Next.js 15.5、Anthropic SDK（Claude Haiku）、@line/bot-sdk、pg 直連、Sharp（圖片）、date-fns、PapaParse、Sentry。
部署：前後台皆部署於 Vercel；GitHub Actions 負責定期備份與 migration 執行。

### 4.4 GB哥（LINE 智能管家）

入口 `api/line/webhook`，核心 `lib/gbBro.ts`。群組訊息需含「gb哥」或「@吉吉比 線上轉蛋」觸發，個人訊息由 `ADMIN_LINE_IDS` 白名單管控，30 分鐘對話記憶，最多 5 輪 tool loop。**執行原則：收到指令立即執行回報，絕不把問題丟回給老闆。** 提供 27 個工具，涵蓋查詢（當收、待辦、庫存、設定、會員、月結、退款、任意可讀 SQL）與寫入（改庫存、售價、上下架、補幣、封帳、取消訂單、核定退款、建立優惠碼、更新文案、記錄功能缺失）。

**v1.1 新增**：
- `@吉吉比 線上轉蛋` 與 `@吉吉比線上轉蛋` 作為 wake word
- Reply threading：管理員回覆 GB哥 的訊息（30 分鐘 TTL，via `gb_sent_messages` 表）可免 wake word 繼續對話

### 4.5 金流與物流（ECPay）

金流：`topup` → 建立 `recharge_records`(pending) → 綠界 AioCheckOut V5 → callback 驗簽（SHA256）、冪等性檢查 → 補 G 幣。物流：後台建立物流單 → CVS 超商選址（HOME 宅配）→ callback（MD5）更新 `tracking_number` 與貨態。冪等性由 `webhook_events` 表保護，同一 ECPay TradeNo 只入帳一次。

## 五、角色分工（以行政組為例）

以下為參考自創辦人規劃的**主責方向**，非固定分工。公司沒有員工只有合夥人，每位行政組都需一起參與，也歡迎跨領域互相補位。主責是為了讓彼此知道聯絡誰。

| 行政組 | 主責 | 對應到平台的模組 |
|---|---|---|
| **潤** | 平台開發、AI 工具、視覺設計、產品規劃、商品上架 | frontend/backend 開發、AI 組織與 GB哥、UI/視覺、產品 roadmap、商品建檔 |
| **采** | 商品管理、廠商合作、採購、供應鏈管理 | products / suppliers / settlement 對接、廠商月結、採購對帳、supply-chain 報告跟進 |
| **泳** | 平台開發、AI 工具輔助、程式協力、平台維護、商品上架 | frontend/backend 開發、cron agents 維護、bug 修復、商品建檔 |
| **音** | 業務開發、對外合作、商品洽詢、客戶關係管理 | KOL/異業合作、廠商與通路洽詢、社群與客戶關係、流量策略落地 |

**合作原則（版友公社精神）**：有話直接說、直接彼此、讓彼此知道一起承擔。有人累了就說出來，我不幫你這個相補，你累了就有人多幫一點，公司的成果是大家一起努力的成果。

## 六、決策與治理機制

**重大決策由合夥行政組共同討論並取得共識**，包括：增資、減資、股本調整、新行政組加入、利潤分配、重大合作案、大額支出、公司方向重大調整等。公司沒有誰說了算。

**制度建立（成立前暫定，成立後正式討論）**：行政組會議、股本比例、退場機制、股份轉讓限制、保密條款、財務透明、版權歸屬、責任人本責、制度不是因為不信任，而是相信大家有一起走長之心，所以更重要先把規矩建立好。

**帳目透明**：收入、支出與帳目交由專業記帳士處理，所有行政組都能瞭解公司的財務狀況。平台本身的 `token_ledger`、月結快照、稽核軌跡（`action_logs`）也提供技術面的透明基礎。

**公司設立**：資本額預估 20 萬，合夥行政組共同出資，負責人由共同討論決定（若由家人代表，需步調與行政組開會協商法律文件）。前期不分紅，到穩定獲利後再討論認股和分紅。

## 七、風險管理

| 風險 | 影響 | 對策 | 主責 |
|---|---|---|---|
| ECPay 仍為測試帳號 | 無法收受實際金流 | 申請正式帳號，換上 Merchant/Hash 環境變數，上線前驗收表 | 潤、采 |
| 退款 API 未整合 | 退款需人工走至綠界操作 | 上線後補 ECPay 退款 API 整合，暫以人工 SOP 替代 | 潤、泳 |
| 後台無自動化測試 | 修改時可能引發無聲迴歸錯誤 | 依測試案例文件補後台核心流程測試 | 泳 |
| 轉蛋殺率未從 DB 動態讀取 | 無法即時調整利潤率 | 讓 drawLogic 讀取 `product_settings.current_profit_rate` | 潤 |
| 預算有限 | 無法大量廣告投放 | 走自然流量策略，AI 內容撰稿降低社群經營成本 | 音、潤 |
| 流量增長遇到 DB 連線瓶頸 | 尖峰時段錯誤 | 適時升級 Supabase/Vercel Pro，必要時 feature flag 停用轉蛋 | 泳 |
| 競品情報依賴外部 API | 報告可能不完整 | 加入備用爬取源與重試邏輯 | 泳 |
| 合夥人認知落差 | 影響合作與共識 | 定期會議、行政組開議，定期溝通 | 全體 |
